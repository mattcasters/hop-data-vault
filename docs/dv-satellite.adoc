= Data Vault Satellite
:toc: macro
:toclevels: 3

toc::[]

A Satellite stores the descriptive, time-varying attributes that belong to a Hub or a Link. Classic examples are customer name and address details, product descriptions and categories, order status history, etc.

In Data Vault 2.0, satellites are insert-only. When any of the tracked attributes changes for a given parent (hub or link), a new row is inserted with the new values and the current load date. Old rows are kept, giving you a complete history.

== How satellite updates work

When data is loaded for a satellite, the generated pipeline:

* Reads the parent business keys (so it can compute the parent hash key), the satellite attributes, and the record source indicator from your source.
* Calculates the parent hash key.
* Compares the incoming data against the existing rows in the satellite table.
* Detects real changes by comparing the actual attribute values (not just the presence of the parent key).
* Inserts only the rows that are new for that parent key or that contain different attribute values.
* Adds the batch load date to every inserted row.

Because the comparison looks at the attribute values themselves, you do not get duplicate "no-change" rows in the satellite. This keeps history clean and query performance good.

== Options in the Satellite dialog

[cols="1,3", options="header"]
|===
|Option |Description

|Name
|The name of this satellite definition. Used as the default physical table name if you do not specify one.

|Physical table name
|The actual table name in the target Data Vault database. Leave blank to use the Name.

|Description
|Business description of the satellite and the kind of information it holds.

|Default record source
|The Data Vault Source that feeds this satellite. Required.

|Parent hub
|Select the Hub this satellite describes (for normal hub satellites). Choose either a parent hub or a parent link, not both.

|Parent link
|Select the Link this satellite describes (for satellites that hold relationship-level attributes). Choose either a parent link or a parent hub, not both.

|Attributes
|The list of descriptive columns whose changes you want to track.

Each attribute has:
* Column name (in both source and target).
* Optional description.
* Data type, length, and precision (used for DDL and loading).
* "Include in CDC" – normally left checked. Uncheck it for columns that should not trigger a new satellite row when they change (for example audit columns or fields added later that you do not want to version).

The **Get Attributes** button is very useful. It populates the list from the referenced source, automatically excluding primary key fields and any driving key. You can then delete columns you do not want to historize or adjust lengths and the "Include in CDC" flag.

If you leave the attributes list empty, the plugin will include all non-key, non-driving-key fields from the source. The model check will warn you about this so you can decide whether it is intentional.

|Multi-active satellite
|Check this box when more than one active record can exist for the same parent key at the same moment in time. A common example is multiple current phone numbers or email addresses for a customer.

|Driving key
|Required when Multi-active is checked. This is the additional column that, together with the parent hash key and load date, makes each row unique (for example the phone number type or the email address itself).

|Driving key source field
|Source column that supplies the driving key value. Renamed to the driving key column in the update pipeline and stored in the target satellite table.
|===

== Checks that involve satellites

The **Check model** button will report problems such as:

* The satellite is not attached to any hub or link.
* No attributes are defined (and therefore all non-key fields from the source will be used).
* One or more attributes have no name.
* The referenced parent hub or link does not exist in the model.
* **Source-to-target type mismatches** for explicit attributes, auto-attributes (when the attribute list is empty), and the driving key when multi-active mode is enabled.

Type validation compares each mapped source column to the satellite target column definition. Errors are raised for incompatible types or when the source is wider than the target. Warnings cover stored-metadata drift from the live source schema.

These checks help you avoid generating pipelines that would fail or produce unexpected results.

== Target table layout for satellites

The layout that drives DDL and loading usually contains:

1. The parent hash key column (the hash key name is taken from the parent hub definition).
2. The satellite attribute columns, in the order you listed them (or all qualifying source columns if the list is empty). Length and precision information from the source definition is used when available so you get realistic column types.
3. The record source column.
4. The load date column.

The same attribute list that is used for change detection is also used for the target table and the final load, so what you see in the dialog is what ends up in the database.

== Tips for satellites

* Use "Get Attributes" and then review the list. Remove columns that are not interesting for history (for example large free-text fields or technical IDs that never change).
* The "Include in CDC" flag is powerful. Use it to ignore columns that would otherwise cause unnecessary new rows (last-updated timestamps from the source, for example).
* For multi-active satellites, be clear about what the driving key represents. It becomes part of the uniqueness rule for the satellite.
* Attach satellites to links when the descriptive information truly belongs to the relationship rather than to one of the participating hubs.
* When the parent hub is fed by multiple record sources, it is common and recommended to create a separate satellite definition for each source (even if they track the same attributes). This keeps the incoming data, record source values, and change history cleanly separated per origin system while still describing the same parent hub.

== Satellites on multi-source hubs

If your hub definition lists several record sources, you will normally also create multiple satellites (one per source) that all point to the same parent hub. Each satellite only needs to reference the Data Vault Source that is relevant to it. This pattern gives you clean per-source history while keeping the hub itself as a single unified business key table.
