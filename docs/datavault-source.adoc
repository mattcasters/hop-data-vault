= Data Vault Source

A Data Vault Source is the logical definition of a source system or data feed that supplies data to your Data Vault model. Hubs, Links, and Satellites refer to Data Vault Sources (in their "Default record source" field) rather than directly to physical connections.

This extra layer gives you a place to:

* Give the source a stable, business-oriented name.
* Define how the record source / source system indicator value should be populated for rows coming from this source.
* Point to the detailed physical definition (today a Data Vault Database Source) that contains the actual connection and column layout.

== Options

[cols="1,3", options="header"]
|===
|Option |Description

|Name
|The name of this source. This is the name you will select when attaching sources to hubs (Record sources tab), links, and satellites (Default record source field). It must be unique within the project.

|Source type
|The kind of source. Currently only "Database" is supported. This field exists so that other source types (files, messages, APIs, etc.) can be added in the future without changing the rest of the model.

|Source table name
|For database sources, this is the name of the Data Vault Database Source metadata object that contains the connection details and column definitions. You pick it from the available database sources in the project.

|Source indicator field
|Optional. The name of a column in the source table that contains, for each row, the value that should be written into the Data Vault RECORD_SOURCE (or equivalent) column.

This field is only used when you do not provide a static Source indicator value. The name you enter here supports variables and will be used (quoted) in the generated source queries.

|Source indicator
|Optional but very commonly used. A fixed value that will be written as the record source for every row coming from this source (for example `ERP`, `CRM`, `ORDERS-2024`, `SALES-FEED`).

When a value is supplied here, it takes precedence over any Source indicator field. The value can contain variables. It is emitted as a SQL string literal in the source query.

This is the simplest and most reliable way to tag data with its origin when the source system itself does not carry a source indicator column.
|===

== How the record source indicator is chosen at runtime

The plugin follows this simple precedence:

1. If the Source indicator field contains a non-empty value (after variable resolution), that static value is used for all rows from this source.
2. Otherwise, if a Source indicator field name is supplied, the generated pipeline reads that column from the source and aliases it to the target record source column name.
3. If neither is provided, no record source indicator is added from this source (a table-level override or a NULL placeholder may still be used for technical reasons during comparison).

You control the target column name for the record source either globally in the Data Vault Configuration or with a per-hub override.

== Why separate Data Vault Source from the physical source definition?

* The physical Data Vault Database Source is about "what columns exist and where do I read them from?"
* The (logical) Data Vault Source is about "this is the feed called SALES-ORDERS and everything from it should be tagged with record source value SALES".

A single physical table can be referenced by multiple logical Data Vault Sources if you need to load it with different record source tags or under different circumstances.

Tables in your model only need to know the logical source name. This keeps the model stable even if the underlying physical details change.

== Validation and checks

When you click **Check model** (or when the Data Vault Update action runs with model checking enabled), the plugin verifies that every table's chosen record source actually exists. A missing reference is reported as an error. This prevents pipelines from being generated against sources that have not been defined yet.

== Practical advice

* Create Data Vault Sources with clear, stable names that reflect the business feed rather than the physical table name.
* Decide early on a convention for record source values (short codes, feed names with dates, system names, etc.) and document it. Consistency here makes auditing and debugging much easier later.
* For sources that truly carry their own source indicator column, use the Source indicator field option. For the majority of cases a static Source indicator value is simpler and more reliable.
* Variables are supported in both indicator fields, so you can make record source values dynamic (for example include a batch date or environment name) if your requirements demand it.
* Keep the physical column layout in the referenced Data Vault Database Source accurate. It drives "Get Keys", "Get Attributes", hash calculations, and target column definitions.

Data Vault Sources are lightweight but important. Good source definitions make the rest of the modeling experience much smoother and the resulting loads more correct.
