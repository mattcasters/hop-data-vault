= Data Vault Source
:toc: macro
:toclevels: 3

toc::[]

A **Data Vault Source** is the logical definition of a source system or data feed that supplies data to your Data Vault model. Hubs, Links, and Satellites refer to Data Vault Sources (in their record source fields) rather than directly to physical database connections.

Each Data Vault Source combines:

* **DV semantics** – How rows from this feed should be tagged with a record source value, and optionally which **group** they belong to for partial workflow loads.
* **Physical source definition** – How to read the data (today an embedded **database** source with connection, table, and field layout; file and API types are planned).

== Editor tabs

The Data Vault Source metadata editor has two tabs:

* **General** – Record source indicator settings and group.
* **Database** – Connection, schema, table, and field layout (see link:datavault-source-database.adoc[Database source details]).

== General tab options

[cols="1,3", options="header"]
|===
|Option |Description

|Name
|The name of this source. Selected when attaching sources to hubs (Record sources tab), links, and satellites (Default record source). Must be unique within the project.

|Source indicator
|Optional fixed value written as the record source for every row from this feed (for example `ERP`, `CRM`, `ORDERS-2024`). Supports variables. When set, takes precedence over Source indicator field.

|Source indicator field
|Optional column in the source that contains the per-row record source value. Used only when Source indicator is empty. Supports variables; the name is quoted in generated SQL.

|Group
|Optional tag used by the **Data Vault Update** action's **Record source group** option. When the action specifies a group, only sources whose Group matches are included in that run. Leave empty to always participate in full loads.
|===

== How the record source indicator is chosen at runtime

The plugin follows this precedence:

1. If **Source indicator** contains a non-empty value (after variable resolution), that static value is used for all rows from this source.
2. Otherwise, if **Source indicator field** is set, the generated pipeline reads that column and aliases it to the target record source column name.
3. If neither is provided, no record source indicator is added from this source (a NULL placeholder may still be used during comparison steps).

The target record source column name comes from the model configuration, unless a hub or link overrides it in its own dialog.

== Why logical and physical details live together

Keeping DV semantics and the database layout in one metadata object means:

* Model tables reference a single stable source name.
* Connection, schema, table, and column metadata travel with the feed definition.
* You can duplicate a Data Vault Source (with a different name, indicator, or group) while reusing the same physical layout when one table is loaded under multiple record source tags.

The **Import** button on the Database tab (and the model toolbar **Import sources** action) reads column metadata from a live database to populate the field list quickly.

== Validation and checks

When you click **Check model** (or when the Data Vault Update action runs with model checking enabled), the plugin verifies that every table's chosen record source exists and that source field definitions are present. A missing reference or empty field list is reported as an error.

When **detailed data type checking** is active, database sources additionally resolve field types from the live table or view (`connection`, `schema`, `table`). The stored field layout on the Database tab remains the baseline for target column definitions and drift detection. If live resolution fails, the check falls back to stored metadata and reports the failure.

Parquet file sources use stored metadata until live field resolution is implemented for that type. Additional file types (planned: Avro, Iceberg) will follow the same pattern.

== Practical advice

* Name sources after the business feed, not only the physical table name.
* Agree on a convention for record source values and document it.
* Use **Group** when you need to split loads (for example hourly vs nightly feeds) without maintaining separate models.
* For sources that carry their own indicator column, use Source indicator field; for most cases a static Source indicator is simpler.
* Keep the Database tab field layout accurate—it drives Get Keys, Get Attributes, hash calculations, and target column definitions.

Data Vault Sources are lightweight but important. Good source definitions make modeling smoother and loads more correct.
