= Data Vault Source
:toc: macro
:toclevels: 3

toc::[]

A **Data Vault Source** is a **Data Catalog record definition** of type `DV_SOURCE`. It describes a logical feed that supplies rows to hubs, links, and satellites. Model tables reference the source **name**, not a raw database connection.

Each definition combines:

* **DV semantics** — record source indicator, optional **group** for partial workflow loads
* **Physical layout** — today typically a **database** table (connection, schema, table, field list); file sources are planned

Sources live under namespace `hop/{project}/sources` in `catalog-data/` and open in the **Data Catalog** perspective. See link:data-catalog.adoc[Data Catalog] for folder layout and refresh behavior.

== Create and edit a source

. Open the Hop project.
. Open the **Data Catalog** perspective.
. Under `hop/{project}/sources`, create or open a `DV_SOURCE` record.
. Set connection, table, and fields (database-backed sources).
. Configure record source indicator and optional group.

The model toolbar **Import sources** action can also create catalog entries from live database metadata.

== JSON structure (file catalog)

Each source is a JSON file. Important fields:

[cols="1,3", options="header"]
|===
|Field |Description

|`namespace`
|Must match folder path (for example `hop/retail-example/sources`)

|`name`
|Unique within namespace; used in hub **Record sources** and satellite **Default record source**

|`type`
|`DV_SOURCE`

|`physicalTable`
|Database connection name, schema, table (`databaseMetaName`, `schemaName`, `tableName`)

|`dvSource.sourceIndicator`
|Optional fixed record source value for every row (for example `CRM`, `ERP`)

|`dvSource.sourceIndicatorField`
|Optional source column for per-row record source (used when indicator is empty)

|`dvSource.group`
|Optional tag for **Record source group** on Data Vault Update

|`dvSource.fields`
|Column layout with types, lengths, primary-key flags

|`rowMetaXml`
|Hop row metadata for pipeline field resolution
|===

Example path: `retail-example/catalog-data/hop/retail-example/sources/E2E-customer-hub.json`.

== Record source indicator at runtime

The plugin applies this precedence:

1. If **source indicator** is non-empty (after variable resolution), that static value is used for all rows.
2. Otherwise, if **source indicator field** is set, the pipeline reads that column and aliases it to the model's record source column name.
3. If neither is set, no indicator is added from this source.

The target column name comes from `.hdv` model configuration unless a hub or link overrides it.

== Database field layout

Database-backed sources store per-column metadata used by **Load from source**, hashing, and type checking. See link:datavault-source-database.adoc[Database source details] for the field editor and **Import** from live schema.

Keeping the field list accurate is essential for **Check model** and **detailed data type checking**.

== Groups and partial loads

Set **group** on a source when only some feeds should run in a batch (for example `nightly` vs `hourly`). On **Data Vault Update**, set **Record source group** to match; sources with a different or empty group are skipped for that run.

== Validation

* **Catalog validation** — feed-level checks, proposals, acknowledgements; see link:resource-definition-validation.adoc[Resource definition validation]
* **Check model** — verifies every referenced source exists and fields are present; optional live schema type checks

When detailed type checking is active, database sources resolve types from the live table. Stored catalog metadata remains the baseline; drift from the live schema produces warnings.

== Practical advice

* Name sources after the business feed, not only the physical table.
* Agree on record source value conventions across the program.
* Use **group** for scheduled partial loads without maintaining separate models.
* Prefer a static source indicator when all rows share one feed identity.
* Refresh the catalog (F5) after editing JSON files on disk.
