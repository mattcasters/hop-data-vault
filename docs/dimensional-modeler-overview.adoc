= Dimensional modeler overview
:toc: macro
:toclevels: 3

toc::[]

The dimensional modeler adds Kimball star/snowflake modeling to the hop-data-vault plugin. Models are saved as `.hdm` files and drive native Hop warehouse transforms for dimension and fact loads.

== Why a separate modeler

Data Vault and Kimball semantics differ enough that they use separate model files:

* `.hdv` ‚Äî hubs, links, satellites, hashing, insert-only history
* `.hdm` ‚Äî dimensions, facts, bridges, grains, measures, SCD policies

The plugins share infrastructure (graph shell, ELK layout, orchestrator, workflow action pattern) but keep domain metadata separate.

== Dimensional model (`.hdm`)

A dimensional model contains:

* **Configuration** ‚Äî target database, standard column names (`dim_key`, `date_from`, `date_to`, ‚Ä¶), pipeline naming
* **Tables** ‚Äî dimensions, facts, junk dimensions, bridges, factless facts, snapshot facts, aggregates
* **Conformed dimension registry** ‚Äî logical name ‚Üí physical dimension table
* **Notes** ‚Äî canvas documentation only

Open a `.hdm` file in Hop GUI for the visual editor (same interaction model as the Data Vault canvas).

=== Table types

[cols="1,3", options="header"]
|===
|Type |Kimball role

|Dimension (`DmDimension`)
|SCD1, SCD2, or SCD3/hybrid dimension loads

|Fact (`DmFact`)
|Transaction fact with dimension roles and measures

|Junk dimension (`DmJunkDimension`)
|Low-cardinality flag combinations via Combination Lookup

|Bridge (`DmBridge`)
|Many-to-many resolution between two or more dimensions

|Factless fact (`DmFactlessFact`)
|Coverage/attendance pattern ‚Äî dimension FKs only

|Periodic snapshot (`DmPeriodicSnapshotFact`)
|Grain includes snapshot period

|Accumulating snapshot (`DmAccumulatingSnapshotFact`)
|Milestone grain with row updates across a lifecycle

|Aggregate fact (`DmAggregateFact`)
|Pre-computed rollup from a base fact
|===

=== Source SQL

Each table carries **source SQL** used by generated `TableInput` transforms. Model validation requires non-empty source SQL before running **Dimensional Update**.

For the standard calendar **`dim_date`** dimension, you can populate the physical table with the link:date-dimension-generator.adoc[Date Dimension Generator] transform instead of hand-written SQL or an external calendar source. The generator's **Load defaults** button matches the attributes on the modeler's date dimension template (`date_key`, `full_date`, `day_of_week`, and related columns).

== Workflow integration

=== Dimensional Update

Production loads use the **Dimensional Update** workflow action (`DIMENSIONAL_UPDATE`):

1. Load `.hdm` model
2. `model.check()` ‚Äî Kimball validation
3. Optional DDL (CREATE TABLE on target database)
4. Per-table `generateUpdatePipelines()` ‚Äî configure native Hop transforms
5. Stage pipelines ‚Üí orchestrate in parallel ‚Üí cleanup

Fixture workflows:

* `project/tests/basic/update-dm-basic-star.hwf` ‚Äî basic star schema
* `project/tests/basic/update-dm-extended-catalog.hwf` ‚Äî full Kimball catalog (pipeline/DDL only)

=== Dimensional Publish

**Dimensional Publish** (`DIMENSIONAL_PUBLISH`) converts a `.hdv` Data Vault model into a **draft** `.hdm` for human review. It does not run ETL.

Fixture: `project/tests/basic/publish-vault1-draft.hwf`

Mapping rules (starting point):

* Hub ‚Üí dimension (business keys ‚Üí natural keys)
* Hub satellite ‚Üí dimension attributes (SCD2 when historized)
* Hub satellite with cross-hub FKs + measures ‚Üí transaction fact
* Link (2 hubs, no measures) ‚Üí bridge (configurable)
* Link (otherwise, no measures) ‚Üí factless fact
* Link satellite with measures ‚Üí fact

Review the draft model in the GUI before running Dimensional Update.

== GUI workflows

=== Canvas context menu

Left-click the dimensional canvas background to:

* **Import database tables** ‚Äî scaffold draft tables from an existing database schema
* Add dimension, fact, junk dimension, bridge, factless fact, or note

=== Data Vault canvas

From a `.hdv` model, use **Publish draft dimensional model** to create a draft `.hdm` via the same publish adapter as the workflow action.

== Pipeline generation strategy

Generators are thin builders over native Hop warehouse transforms:

[cols="2,2,3", options="header"]
|===
|Kimball pattern |Hop transform |Notes

|SCD Type 1 |Insert/Update |Overwrite attributes on natural key match

|SCD Type 2 |MergeRows chain (satellite pattern) |Ported from `DvSatellite` load-end-date chain

|SCD Type 3 / hybrid |Dimension Lookup (`update=true`) |Per-attribute update types

|Junk dimension |Combination Lookup |Key field combinations ‚Üí surrogate key

|Fact FK resolution |Dimension Lookup (`update=false`) |One lookup per dimension role

|Bridge / factless / snapshots |Mix of above + TableOutput/InsertUpdate |Catalog-specific builders
|===

== Database import heuristics

**Import database tables** infers table type from naming:

* `d_*`, `dim_*` ‚Üí dimension
* `f_*`, `fact_*` ‚Üí fact
* `bridge_*` ‚Üí bridge
* `factless_*`, `f_factless_*` ‚Üí factless fact

Column heuristics:

* Skip standard warehouse columns (`dim_key`, `version`, `date_from`, ‚Ä¶)
* `*_id` / `*_code` ‚Üí natural key candidate on dimensions
* Integer `*_key` columns ‚Üí dimension roles on facts/bridges/factless facts
* Numeric columns ‚Üí measures on facts

Imported tables are **draft** metadata ‚Äî review grains, roles, and source SQL before production loads.
