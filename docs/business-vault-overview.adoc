= Business Vault overview
:toc: macro
:toclevels: 3

toc::[]

The Business Vault layer sits on top of the raw Data Vault. It turns insert-only satellite history into consumption-ready tables — primarily **functional SCD Type 2** timelines and **point-in-time (PIT)** snapshot helpers for technical as-of queries.

Business Vault models are saved as `.hbv` files and always reference a linked `.hdv` Data Vault model.

== Why Business Vault

Raw Data Vault satellites record **when the warehouse loaded** a change (technical load timestamp). Reporting and operational analytics often need **when the source system said** a value was true (functional effective date).

Hand-building SCD2 or as-of logic in SQL or dbt for every satellite is repetitive and drifts from the vault model. The plugin generates Hop pipelines from DV metadata:

* **SCD2** — reads satellite history, collapses versions into `valid_from` / `valid_to` intervals, applies open-ended sentinels, loads a BV table. SCD2 is **pure Type 2**; hybrid Type 1 + Type 2 presentation and “always current” attributes are expected in dimensional models or via a separate current-state join, not by rewriting SCD2 history (see link:business-vault-scd2.adoc#type-1-vs-type-2-and-hybrid-attributes[Type 1 vs Type 2 and hybrid attributes]).
* **PIT** — reads raw hub and satellite history, materializes snapshot dates with satellite load-timestamp pointers for fast as-of joins (not hybrid attribute storage)

== Business Vault model (`.hbv`)

A Business Vault model contains:

* **Data Vault model path** — path to the linked `.hdv` file (variables such as `${PROJECT_HOME}` supported)
* **Configuration** — target database, SCD2 defaults, pipeline naming; see link:business-vault-configuration.adoc[Business Vault Configuration]
* **Tables** — SCD2 tables, PIT tables, and future business table types

Open a `.hbv` file in Hop GUI to get a visual editor comparable to the Data Vault canvas.

image::images/business-vault-model-customer-360.png[Business Vault model canvas — Customer 360 SCD2 table,align="center"]

=== Canvas and toolbar

The Business Vault editor uses the same touch-friendly interaction model as the Data Vault editor:

* **Left-click context menus** with icon actions (no right-click)
* Drag tables to position them
* Toolbar: **Edit model**, **Check model**, **Reload DV model**, **Debug** (generate build pipelines), zoom

=== Navigate to raw vault tables

From a Business Vault table you can jump to the linked Data Vault model and select the referenced hub, link, or satellite. This uses the path configured on the `.hbv` file and requires `${PROJECT_HOME}` (or an absolute path) to resolve correctly.

== Table types

[cols="1,3", options="header"]
|===
|Type |Purpose

|SCD2 (`BvScd2Table`)
|Merge one or more DV satellites into a functional SCD2 table. See link:business-vault-scd2.adoc[Business Vault SCD2].

|PIT (`BvPitTable`)
|Point-in-time helper for technical as-of queries: hub grain × snapshot dates with satellite load-timestamp pointers. See link:business-vault-pit.adoc[Business Vault PIT].

|Business / SQL table (`BvBusinessTable`)
|SQL-sourced object materialised as a **view** or **table** (`CREATE OR REPLACE VIEW|TABLE … AS …`) with dbt-style `{{ ref() }}` / `{{ source() }}` dependencies. Multi-step layers can `ref()` tables in another `.hbv` or place **Business Vault references** on the canvas. See link:business-vault-sql-view.adoc[Business Vault SQL views and tables].

Data Vault hubs, links, and satellites are referenced with **canvas aliases** (Add Hub / Link / Satellite reference), optionally from multiple `.hdv` models via the catalog or file browse. There is no required single “linked Data Vault model file” on the BV.
|===

== Workflow integration

Production rebuilds use the **Business Vault Update** workflow action (`BUSINESS_VAULT_UPDATE`). It mirrors the Data Vault Update pattern: model check, optional DDL, pipeline generation, parallel orchestration, optional catalog publish.

See link:business-vault-update-action.adoc[Business Vault Update Action].

== Relationship to raw Data Vault

```mermaid
flowchart LR
  Hdv[".hdv Data Vault model"]
  VaultDB["Vault database"]
  Hbv[".hbv Business Vault model"]
  BvDB["BV target database"]
  Hdv --> VaultDB
  Hdv --> Hbv
  Hbv --> VaultDB
  Hbv --> BvDB
```

Business Vault **reads** hub and satellite history from the Data Vault target database (configured on the `.hdv` model). It **writes** SCD2 and PIT tables to the Business Vault target database (configured on the `.hbv` model). Both connections must exist in Hop project metadata. SCD2 and PIT are independent data paths; Business Vault Update runs SCD2 build pipelines first, then PIT.

== External Data Vault tables

When raw vault tables are loaded outside Hop (dbt, SQL, custom ETL), set **Integration mode** to **External read-only** on the DV table. Business Vault SCD2 generation still works as long as the `.hdv` metadata accurately describes the warehouse tables.

See link:dv-integration-modes.adoc[Data Vault integration modes].
