= Business Vault overview
:toc: macro
:toclevels: 3

toc::[]

The Business Vault layer sits on top of the raw Data Vault. It turns insert-only satellite history into consumption-ready tables — primarily **functional SCD Type 2** timelines with standardized column names.

Business Vault models are saved as `.hbv` files and always reference a linked `.hdv` Data Vault model.

== Why Business Vault

Raw Data Vault satellites record **when the warehouse loaded** a change (technical load timestamp). Reporting and operational analytics often need **when the source system said** a value was true (functional effective date).

Hand-building SCD2 logic in SQL or dbt for every satellite is repetitive and drifts from the vault model. The plugin generates Hop pipelines from DV metadata:

* Reads satellite history from the vault database
* Collapses attribute versions into `valid_from` / `valid_to` intervals
* Applies open-ended sentinels for current rows
* Loads a physical Business Vault table

== Business Vault model (`.hbv`)

A Business Vault model contains:

* **Data Vault model path** — path to the linked `.hdv` file (variables such as `${PROJECT_HOME}` supported)
* **Configuration** — target database, SCD2 defaults, pipeline naming; see link:business-vault-configuration.adoc[Business Vault Configuration]
* **Tables** — SCD2 tables, PIT tables, and future business table types

Open a `.hbv` file in Hop GUI to get a visual editor comparable to the Data Vault canvas.

image::images/business-vault-model-customer-360.png[Business Vault model canvas — Customer 360 SCD2 table,align="center"]

=== Canvas and toolbar

The Business Vault editor uses the same touch-friendly interaction model as the Data Vault editor:

* **Left-click context menus** with icon actions (no right-click)
* Drag tables to position them
* Toolbar: **Edit model**, **Check model**, **Reload DV model**, **Debug** (generate build pipelines), zoom

=== Navigate to raw vault tables

From a Business Vault table you can jump to the linked Data Vault model and select the referenced hub, link, or satellite. This uses the path configured on the `.hbv` file and requires `${PROJECT_HOME}` (or an absolute path) to resolve correctly.

== Table types

[cols="1,3", options="header"]
|===
|Type |Purpose

|SCD2 (`BvScd2Table`)
|Merge one or more DV satellites into a functional SCD2 table. See link:business-vault-scd2.adoc[Business Vault SCD2].

|PIT (`BvPitTable`)
|Point-in-time helper table referencing a hub and one or more satellites. Pipeline generation follows the PIT prefix configured on the model.

|Business table (future)
|Placeholder type for non-SCD2 consumption tables.
|===

== Workflow integration

Production rebuilds use the **Business Vault Update** workflow action (`BUSINESS_VAULT_UPDATE`). It mirrors the Data Vault Update pattern: model check, optional DDL, pipeline generation, parallel orchestration, optional catalog publish.

See link:business-vault-update-action.adoc[Business Vault Update Action].

== Relationship to raw Data Vault

```mermaid
flowchart LR
  Hdv[".hdv Data Vault model"]
  VaultDB["Vault database"]
  Hbv[".hbv Business Vault model"]
  BvDB["BV target database"]
  Hdv --> VaultDB
  Hdv --> Hbv
  Hbv --> VaultDB
  Hbv --> BvDB
```

Business Vault **reads** satellite history from the Data Vault target database (configured on the `.hdv` model). It **writes** to the Business Vault target database (configured on the `.hbv` model). Both connections must exist in Hop project metadata.

== External Data Vault tables

When raw vault tables are loaded outside Hop (dbt, SQL, custom ETL), set **Integration mode** to **External read-only** on the DV table. Business Vault SCD2 generation still works as long as the `.hdv` metadata accurately describes the warehouse tables.

See link:dv-integration-modes.adoc[Data Vault integration modes].
