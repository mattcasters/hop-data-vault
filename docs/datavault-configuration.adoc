= Data Vault Configuration

A Data Vault Configuration holds the technical and naming decisions that affect how your Data Vault tables are created and loaded. By keeping these decisions in a separate, reusable metadata object you can apply the same rules across many hubs, links, and satellites, while still being able to override them for individual tables when needed.

You normally create one main configuration for a model or subject area and reference it from the Data Vault Model. Individual tables can point to a different configuration if they have special requirements.

== Target database

[cols="1,3", options="header"]
|===
|Option |Description

|Target database
|The Hop database connection (DatabaseMeta) where the Data Vault tables will live. This connection is used both to read source data (through your Data Vault Sources) and to write to the target tables.

The plugin uses the connection's dialect for quoting identifiers when it builds SQL, so the generated pipelines work correctly with your particular database (PostgreSQL, SQL Server, Snowflake, etc.).
|===

== Hashing strategy

These settings control how surrogate hash keys are calculated and stored.

[cols="1,3", options="header"]
|===
|Option |Description

|Hash algorithm
|The algorithm used to turn business keys (and link combinations) into deterministic surrogate keys.

Common choices:
* MD5 – Fast and widely used. Good default for most projects.
* SHA-256 – Larger key, lower theoretical collision risk for very large data volumes.
* SHA-1 and SHA-512 are also available.

The chosen algorithm affects both the computation and the maximum length of the hash key columns in the target tables.

|Hash key data type
|How the hash key values are physically stored in the database.

* BINARY (recommended for production) stores the raw bytes. It uses the least space and avoids collation or character-set issues when joining or comparing keys.
* HEX stores a hexadecimal string (for example 32 characters for an MD5). Easy to read and debug, but uses more storage.
* STRING stores the hash in a decimal-dash format produced by some checksum implementations. Mainly useful for compatibility or debugging.

Choose BINARY for new projects unless you have a strong reason to store the keys in a human-readable form.

|Hash content casing
|Controls whether business key values are upper-cased, lower-cased, or left as-is before hashing. UPPER is the most common choice in Data Vault practice because it produces consistent keys even when source systems use mixed case.

|Business key delimiter
|A string used to concatenate the parts of composite business keys (and when building link hash inputs). Pick a delimiter that is very unlikely to appear inside your actual key values (examples: `||`, `|`, `~`, `^`).

|Null placeholder
|The value that replaces NULL or missing key parts during hashing. Using a consistent placeholder ensures that two rows with the same "missing" parts produce the same hash key. A common choice is `^^`.

|Trim business keys
|When enabled (the default), leading and trailing spaces are removed from business key values before hashing. This is almost always desirable.
|===

== Unknown / ghost record handling

[cols="1,3", options="header"]
|===
|Option |Description

|Generate unknown record
|When checked, the model expects a special "unknown" or "ghost" record in hubs (and corresponding entries in links) for business keys that are missing or not yet mapped. This is a very common Data Vault pattern that lets you load facts or relationships even when the related hub row does not exist yet.

|Unknown business key value
|The business key value that represents "unknown" (for example `N/A`, `-1`, or `UNKNOWN`). This value is used when the special unknown hash key record is created.
|===

== Naming conventions

[cols="1,3", options="header"]
|===
|Option |Description

|Link hash key suffix
|Used as a fallback when deriving a link hash key column name. Most teams set an explicit "Link hash key field name" on the link definition instead, so this setting is rarely needed.

|Load date field name
|The column name used for the load timestamp in all Data Vault tables (typically LOAD_DATE or LDTS). Every generated update pipeline adds a Constant transform that writes the same load date value to every row loaded in that batch.

|Load end date field name
|Column name for an optional load end date (used by teams that prefer an end-dated satellite pattern). The standard generated pipelines are insert-only; this field is available for documentation or custom extensions.

|Record source field name
|The column that records where each row originally came from (commonly RECORD_SOURCE or RSRC). This name is used in the target tables and in the source queries that feed the load pipelines.

|Record source field length
|The length (in characters) of the record source column in the target tables. Supports variables. Default is 100 if nothing is specified.

|Source field name
|An optional secondary source identifier column. Use it if your architecture needs an extra level of source granularity beyond the main record source.
|===

== Satellite loading pattern

[cols="1,3", options="header"]
|===
|Option |Description

|Use load end date pattern
|Indicates that satellites in this configuration are expected to use an end-dating approach rather than (or in addition to) pure insert-only with change detection. The current generated pipelines perform change detection by comparing attribute values; this flag is mainly for documentation and future use.
|===

== Overriding the configuration on individual tables

In the dialog for a Hub, Link, or Satellite you will see a "Configuration" picker. When you select a configuration there, that table will use the chosen settings instead of the one referenced by the Data Vault Model. This is useful when:

* One particular hub needs a longer record source column.
* A satellite should use a different hash key data type for easier debugging.
* A link requires a specific hash algorithm for integration with another system.

Most of the time you can leave the table-level configuration empty and let the model-level default apply everywhere.

== How the configuration affects daily work

* The target database connection is required for the Data Vault Update action and for the Debug feature to generate working pipelines.
* Hash algorithm and data type directly influence storage size and whether people can easily read the hash keys when querying the vault.
* Naming settings (LOAD_DATE, RECORD_SOURCE, etc.) should be agreed early with the team that will consume the Data Vault so that everyone uses the same column names.
* The "Check model" function will warn you if no configuration is referenced at the model level, because many downstream steps (DDL, hashing, column naming) depend on it.

Create the configuration early in a project. It becomes the single place where the technical "rules of the road" for your Data Vault are documented and enforced.
