Day-to-day operation of hop-datavault loads: environments, Docker runners, batch orchestration, and partial model updates.

== Docker runners

From the repository root:

[source,bash]
----
# PostgreSQL on port 54320
./scripts/run-postgres.sh up

# Run a workflow in a Hop container
./scripts/run-hop.sh retail-example workflows/run-retail-initial.hwf
./scripts/run-hop.sh integration-tests tests/run-tests.hwf
----

See link:../scripts/README.md[scripts/README.md] and link:../integration-tests/PROJECT.md[integration-tests/PROJECT.md] for multi-database CI runs.

== Batch orchestration pattern

Typical production workflow chain:

1. **Data Vault Update** on `.hdv` — raw vault load (skips external read-only tables)
2. **Business Vault Update** on `.hbv` — rebuild SCD2 / PIT tables
3. **Dimensional Update** on `.hdm` — load marts

Configure on each action:

* Pipeline run configuration (connection, logging, variables)
* **Parallel pipeline copies** — throughput vs resource use
* DDL options — execute on database, export to file, or fail if DDL needed (CI gate)
* Optional **catalog publish** after DV/BV updates

Retail example: `retail-example/workflows/run-retail-initial.hwf` and `run-retail-update.hwf`.

== Partial loads (record source groups)

Tag `DV_SOURCE` definitions with a **group** (for example `hourly`, `daily`). On **Data Vault Update**, set **Record source group** to load only matching sources. Empty group means all sources participate.

See link:datavault-source.adoc[Data Vault Source] and link:datavault-update-action.adoc[Data Vault Update action].

== Metrics and regression

The integration-tests project collects Data Vault / Business Vault update metrics JSON and runs golden-dataset unit tests. Use it as a reference for CI gates (model check abort, fail-if-DDL-needed).

=== Workflow load overview

Bracket DV/BV/DM update waves with two workflow actions:

1. **Begin Vault Update** — assigns `DV_WORKFLOW_EXECUTION_ID` (and `DV_WORKFLOW_EXECUTION_STARTED_AT`) before model updates run.
2. **End Vault Update** — correlates all `load_run` rows with that execution id, publishes `workflow_load_overview` / `workflow_load_overview_model` to the OPS database, and optionally writes Markdown/HTML reports and logs a full overview.

Retail example (`run-retail-update.hwf`):

```
Load CSV to CRM → Begin Vault Update → update-retail-dv-bv-dm → End Vault Update → Advance load control
```

Configure **End Vault Update** with the same execution metrics profile as the update actions (`retail-execution-metrics`). Report files default to `${PROJECT_HOME}/reports`.

Catalog record definitions live under `catalog-data/hop/<project>/operations/` (`workflow_load_overview`, `workflow_load_overview_model`).
