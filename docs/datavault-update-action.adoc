= Data Vault Update Action

The **Data Vault Update** workflow action is the main way to load a complete Data Vault model in production. You point it at a saved Data Vault Model (`.hdv` file), tell it which pipeline run configuration to use, and it takes care of generating and executing the update pipelines for every hub, link, and satellite in the model.

== Options in the action dialog

[cols="1,3", options="header"]
|===
|Option |Description

|Action name
|The name that appears in the workflow.

|Data Vault Model file
|Path to the `.hdv` file that contains your Data Vault Model. You can use variables. The action loads the model from this file at runtime.

|Pipeline run configuration
|The name of a Pipeline Run Configuration that will be used to execute every generated update pipeline. This setting is required. Choose (or create) a run configuration that has appropriate database connections, parallelism, and logging settings for Data Vault loads.

|Log model check failures
|When enabled (default is on), the action runs the model's validation checks before generating any pipelines. Any warnings or errors are written to the log using logBasic. This gives you visibility into problems without necessarily stopping the load.

|Abort on model check failures
|When enabled (default is on), the action runs the same checks. If any errors are found, it writes an error message and stops immediately without generating or running any update pipelines. This is a safety mechanism so that a broken or incomplete model does not produce a partial or incorrect Data Vault load.
|===

== What happens when the action runs

1. The model is loaded from the `.hdv` file.
2. If either of the model check options is turned on, the action performs the same validations that the "Check model" toolbar button uses in the visual editor.
3. Any warnings or errors are logged (when logging is enabled).
4. If abort is enabled and errors were found, the action fails and returns without doing any loading.
5. A single load date (the current date and time) is captured. This same value is used for every table loaded in this execution of the action.
6. For each table defined in the model the action:
   * Generates the update pipeline(s) for that table (hubs, links, and satellites that have multiple record sources will produce one pipeline per source).
   * Executes the pipeline(s) using the chosen run configuration.
   * Waits for completion of each.
   * Adds the row counts (read, written, input, output, updated, deleted) and any errors from the pipelines to the overall action result.
7. At the end, the action succeeds only if every table completed without errors.

Because one load date is used for the whole batch, every row written during this run of the action will have the same LOAD_DATE value. This makes it easy to identify what was processed together and supports restartability and auditing.

== Model validation inside the action

The checks performed when "Log model check failures" or "Abort on model check failures" are enabled are the same rich set of validations that the visual model editor provides. They look for:

* Missing or incomplete business keys on hubs.
* Satellites that are not attached to any hub or link.
* Links that do not have at least two participating hubs.
* References to record sources, hubs, or links that do not exist in the model.
* Duplicate names for tables or sources.
* Missing Data Vault Configuration on the model (treated as a warning).

These checks are deliberately thorough. They surface many problems that would otherwise only appear as confusing runtime errors or incorrect data in the vault.

You can leave both check options enabled in production. The combination of "log but continue" and "abort on real errors" gives you good visibility while still protecting the vault from obviously broken models.

== Typical usage patterns

* Place the Data Vault Update action after any steps that prepare or land the source data that the model depends on.
* Use a dedicated pipeline run configuration for Data Vault loads so you can tune database connections, rowset sizes, and parallelism independently of other work.
* In most cases leave both model-check options enabled. A model that fails its own checks should normally stop the workflow rather than load partial results.
* If you have a very large model, you can split it across several `.hdv` files and several Data Vault Update actions if you need more granular control or parallel loading.
* The action participates in normal workflow logging, error handling, and retry logic. You can use workflow variables to parameterize the model file name or run configuration.

== Relationship with the visual model editor

The Debug button in the model editor and the Data Vault Update action use the same underlying generation logic. The main differences are:

* Debug opens the generated pipeline(s) in the Hop GUI for inspection or manual execution.
* The action runs the pipelines headlessly under a chosen run configuration and aggregates results.
* The action adds the optional model-check gate and the single batch load date.

This consistency means that what you see when you debug a table is essentially what will happen when the action loads the whole model.

== Troubleshooting

If the action fails:

* Look at the log for model check messages first (when the check options are on).
* Individual pipeline errors are logged with the name of the table that failed.
* Row counts from all pipelines are accumulated even when some tables fail, which can help you see how far the load got.

Because the model file is loaded at runtime, make sure the `.hdv` file is accessible from the environment where the workflow runs (use a relative path inside the project or a variable that points to a stable location).
