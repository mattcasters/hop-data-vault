= Data Vault Update Action

== Description

The **Data Vault Update** workflow action is the main way to load a complete Data Vault model in Apache Hop. You point it at a saved Data Vault Model (`.hdv` file), choose a pipeline run configuration, and the action takes care of the rest.

At runtime the action:

* Loads the model from the `.hdv` file.
* Optionally validates the model before any work is done.
* Optionally generates DDL (CREATE/ALTER statements) for the target database tables defined in the model. DDL handling is independent of the data load: you can execute it on the database, write it to a SQL file, require that no DDL is needed, or combine these options.
* Generates and executes update pipelines for every hub, link, and satellite in the model (unless you choose to skip the data load).
* Uses a single load date for the entire batch, so every row written during one run of the action receives the same `LOAD_DATE` value.
* Aggregates row counts and errors from all generated pipelines into the workflow action result.

The action uses the same pipeline generation logic as the **Debug** button in the visual Data Vault model editor. What you inspect in the GUI is what runs in the workflow, with the addition of batch load date, optional model checks, and flexible DDL handling.

== Options

The following options are available in the action dialog.

=== Action name

The name of the action as it appears in the workflow. This is a standard Hop workflow setting and does not affect how the Data Vault is loaded.

=== Data Vault Model file

Path to the `.hdv` file that contains your Data Vault Model. Variables are supported (for example `${PROJECT_HOME}/vault1.hdv`). The action loads the model from this file at runtime. This setting is required.

=== Pipeline run configuration

The name of a Pipeline Run Configuration used to execute every generated update pipeline. This setting is required whenever the action runs data load pipelines. Choose a run configuration with the appropriate database connections, parallelism, and logging settings for your Data Vault loads.

Use the icons next to the field to run, create, or manage pipeline run configurations directly from the dialog.

=== Log model check failures

When enabled (default: on), the action runs the same validation checks as the **Check model** button in the visual model editor before generating pipelines or DDL. Any warnings or errors are written to the log. This gives you visibility into model problems without necessarily stopping the load.

=== Abort on model check failures

When enabled (default: on), the action runs the same model checks. If any errors are found, it logs an error message and stops immediately without generating DDL or running any update pipelines. Warnings alone do not cause the action to abort.

Use this together with **Log model check failures** to protect the vault from obviously broken or incomplete models.

=== Update target database structure

When enabled (default: on), the action generates CREATE/ALTER DDL for every table in the model and executes it on the target database(s) before running the data load pipelines.

Disable this option when you manage target schemas outside of this action—for example with a separate migration tool, hand-written SQL, or pre-provisioned tables. You can still capture the DDL in a file using **DDL SQL filename** without executing it.

=== DDL SQL filename

Optional path to a SQL file where generated DDL is written when schema changes are needed. Variables are supported. When the model spans multiple target databases, the file contains sections headed by `-- Target database: <connection name>` followed by the DDL statements for that database.

DDL is written to this file only when the action determines that schema changes are actually required. You can use this option on its own (without **Update target database structure**) to export DDL for review or for use in an external deployment process.

=== Fail if DDL is needed

When enabled, the action fails immediately if any DDL would be required to bring the target database structure in line with the model. The failure happens before DDL is executed or written to a file.

Use this as a guard in production or CI/CD workflows to ensure that target schemas are already up to date before data is loaded. If no DDL is needed, the action continues normally.

=== Do not update target database

When enabled, the action skips running the generated data load pipelines. DDL generation, file export, and execution (when those options are configured) still run.

Typical uses include exporting or applying DDL without loading data in the same workflow step, or running schema maintenance separately from the data load.
