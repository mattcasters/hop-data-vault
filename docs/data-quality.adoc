= Data quality measure and quality gate
:toc: macro
:toclevels: 3

toc::[]

Protect loads and observe warehouses with a reusable **measure → disposition** stack. Schema contract validation (resource definition validation) remains separate; this layer checks **data content** (row counts, domains, ranges, nulls).

== Measure vs disposition

| Stage | Responsibility | Workflow fail on findings? |
|-------|----------------|----------------------------|
| **Measure Data Quality** action | Run rules against catalog subjects; produce an ephemeral report | No (only infrastructure errors) |
| **Evaluate Quality Gate** action | Apply policy to a report | Yes for `FAIL_ON_BLOCKING` / `FAIL_ON_WARNINGS`; no for `ALERT_ONLY` |

Pre-update (protect the EDW):

----
Measure Data Quality (sources)
        |
        v
Evaluate Quality Gate (FAIL_ON_BLOCKING) --fail--> stop
        |
      success
        v
Data Vault Update / other load
----

Post-update (observe only; cannot un-load):

----
… load …
        |
        v
Measure Data Quality (target tables, lifecycle=POST_UPDATE)
        |
        v
Evaluate Quality Gate (ALERT_ONLY) --> log findings, continue
----

== Rules

Central **Data quality rule set** Hop metadata holds reusable rules. Catalog **record definitions** bind rules (library reference or inline) via `qualityRules`.

Phase 1 rule types:

* `MIN_ROW_COUNT` / `MAX_ROW_COUNT` — dataset
* `NOT_NULL` — field
* `ALLOWED_VALUES` — field (`values=M,F,U`, `nullAllowed=true`)
* `RANGE` — field (`min`, `max`)
* `NOT_EMPTY_STRING` — field

Parameters use `key=value` pairs (semicolon-separated in the rule set editor).

== Catalog bindings

On a record definition JSON document:

[source,json]
----
"qualityRules": [
  {
    "inlineRule": {
      "id": "not-empty",
      "name": "Table not empty",
      "type": "MIN_ROW_COUNT",
      "severity": "BLOCKING",
      "enabled": true,
      "parameters": { "min": "1" }
    },
    "enabled": true
  }
]
----

Or reference a rule set:

[source,json]
----
"qualityRules": [
  {
    "ruleSetName": "retail-domains",
    "ruleId": "gender-allowed",
    "fieldNameOverride": "gender",
    "enabled": true
  }
]
----

== Workflow actions

=== Measure data quality

* Scope: resource definition group, and/or `namespace/name` keys, and/or namespace prefix
* Evaluation mode: `AUTO` (SQL pushdown for databases in Phase 1)
* Lifecycle: `PRE_UPDATE`, `POST_UPDATE`, `AD_HOC`
* Logs a formatted report; stores it on the workflow for the next gate step

=== Evaluate quality gate

* Disposition: `FAIL_ON_BLOCKING`, `FAIL_ON_WARNINGS`, `ALERT_ONLY`
* Uses the prior measure report when available; otherwise re-measures if configured

== Phase 1 limits

* Database physical tables via SQL pushdown are fully supported
* File/Iceberg sample scanning is deferred
* Profile snapshots are ephemeral (no history store yet)
* Catalog **Quality** tab shows bindings and **Test measure**; authoring is via rule-set metadata and/or JSON bindings on the record (rich inline editor later)

== Retail example

The `retail-example` project ships a ready-made library and wiring:

| Asset | Location |
|-------|----------|
| Rule set metadata | `retail-example/metadata/data-quality-rule-set/retail-source-quality.json` |
| Source bindings | `qualityRules` on each `E2E-*` catalog source (generated by `scripts/generate-catalog-sources.py`) |
| Pre-load gate | `Measure source data quality` → `Evaluate source quality gate` (`FAIL_ON_BLOCKING`) |
| Post-load alert | `Measure source data quality (post)` → `Alert on source quality (post)` (`ALERT_ONLY`) |

Workflows: `run-retail-initial.hwf`, `run-retail-update.hwf`.

Sample rules include table-not-empty, `customer_id` / `product_name` not null, segment / loyalty / order status / channel domains, quantity and demo_score ranges.
