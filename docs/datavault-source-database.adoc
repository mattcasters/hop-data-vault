= Data Vault Source (Database)

A Data Vault Database Source describes a table, view, or queryable object in a source system. It tells the Data Vault model what columns are available, which of them form the natural or primary key, and what their data types and lengths are.

This information is used in several important ways:

* The "Get Keys" button in the hub dialog (Keys tab) and the "Get Attributes" button in the satellite dialog use it to populate business keys and attributes. For hubs with multiple sources, Get Keys reads from the source currently selected in the hub's Record sources list.
* Pipeline generation uses the primary key flags to decide which columns to read for hash calculations.
* Target table layouts (and therefore the DDL that creates your Data Vault tables) use the length and precision information so you get realistic column definitions instead of overly generic ones.

== Options

[cols="1,3", options="header"]
|===
|Option |Description

|Name
|The name you give this source definition inside your project. Other objects (Data Vault Sources, Hubs, Links, Satellites) will refer to it by this name.

|Description
|Optional free-text description of the source system or data feed. Useful for documentation.

|Database connection
|The Hop database connection (DatabaseMeta) that points to the source system. This is required for the Import button and for the generated pipelines to know where to read data from.

|Schema
|Optional schema or catalog name that qualifies the table in the source database.

|Source table / view
|The physical table or view name in the source database. Can be left empty if the actual table name will be supplied at runtime (via variables or a parameter) or if you only use this source definition for its column layout.

|Fields (row layout)
|The list of columns that you expect to come from this source. Each field includes:

* Field name (exactly as it appears in the source).
* Optional description.
* Source data type (the database type such as VARCHAR, NUMBER, DATE, etc.).
* Length and precision.
* "Part of source primary key" flag – this is the most important piece of information for hubs and satellites. Fields marked as primary key become the business keys for hubs and the input used to calculate parent hashes for satellites and links.
* Hop data type (used internally when building row layouts).

The Fields grid is the single source of truth for column metadata that flows through the entire model.
|===

== The Import button

The easiest way to fill in the Fields list is to use the **Import** button in the editor.

1. Select or enter a database connection.
2. Optionally enter a schema and table name.
3. Click Import.

The editor connects to the database, reads the column metadata (names, types, lengths, precision), and tries to detect the primary key columns. It then replaces the current Fields list with the imported information.

You can still edit the grid afterwards. Any manual changes you make are saved with the metadata object and become the authoritative definition for the model, even if the underlying source table changes later.

== How the source definition is used during loading

When a hub, link, or satellite needs to read data:

* It follows the chain from its "Default record source" → the Data Vault Source → the Data Vault Database Source.
* Primary-key fields declared here are used to build the SELECT list for the source query and to decide what goes into hash calculations.
* Length and precision values are copied into the target table layout. This is why you often see proper VARCHAR(50) or DECIMAL(15,2) columns created in your Data Vault instead of TEXT or unlimited strings.

Keeping the source definitions accurate and up to date is one of the highest-leverage things you can do when working with the plugin.

== Relationship with Data Vault Source

A Data Vault Database Source contains the physical connection and column details.

A (logical) Data Vault Source wraps it, adds the record source indicator settings (static value or source column), and is what hubs, links, and satellites actually reference in their dialogs.

This separation keeps the model independent of the concrete source technology. Today only database sources are supported, but the structure allows other source types to be added later without changing your hub, link, and satellite definitions.

== Tips

* Import early and review the primary key flags. Incorrect primary key information is the most common cause of wrong hash keys or missing business keys.
* Adjust lengths and precisions in the source definition if the live database metadata is not quite what you want in the target Data Vault (for example you may want to be more conservative on string lengths).
* Use meaningful names for your Data Vault Database Source objects. They often correspond to source tables or subject areas and will be referenced from many places.
* If the same physical table is used by multiple logical feeds that need different record source values, create one Database Source for the column layout and multiple (logical) Data Vault Sources that point to it with different indicator settings.
